# forge-wiki

**[English](README.md)** · **[한국어](README.ko.md)** · 日本語 · **[中文](README.zh.md)**

LLMエージェント — そして共に働く人間 — のためのガバナンス内蔵マークダウンWiki。

エージェント知識ツールの多くは、完全自動メモリ(mem0、Zep、Letta — 承認という概念が
存在しない)か、人間によるキュレーションサービスかの両極端。forge-wikiはその中間を守る:
**エージェントが書き、人間がゲートし、インデックスは自己維持される。**

## 3つの設計コミットメント

1. **コンテンツはキュレーション、新しさは導出。** `INDEX.md`の`AUTO-INDEX`ブロックは
   フロントマターとファイル名日付から再生成される — このブロックのマージ衝突は手作業では
   なく再生成で解消される。並行マルチライターシミュレーションで検証済み(`tests/`参照 —
   最大50ライター、破壊注入込み)。
2. **OKF v0.1互換。** ファイルはマークダウン + YAMLフロントマター(`type`、`description`)、
   予約`index.md`、リンク=関係エッジ — どのOKFコンシューマでも読め、ロックインなし。
3. **スコープ書き込み、ゲート付きマージ。** 各ライターは自分のファイルのみを所有し、
   他者ファイルの変更は静かな編集ではなく提案(PR/ハンドオフ)。HITLは機能フラグではなく
   書き込みプロトコルそのもの。ヒューマンゲートは**好み** — 個人・組織の判断 — を載せる
   場所であり、欠陥を捕まえる場所ではない: 機能的完全性は機械層の責任で、レビュアーに
   届く前にシミュレーションで検証される。

## クイックスタート

インストール不要: Pythonファイル1つ、`python3`以外の依存なし。一度cloneしてどこに置いても
よく、パスで呼び出す。

```sh
git clone https://github.com/chrono-meta/forge-wiki.git ~/tools/forge-wiki
alias fw='python3 ~/tools/forge-wiki/bin/fw.py'   # 任意 — 以下の例はすべて`fw`を使う

cd your-knowledge-repo   # Wikiを格納するリポジトリ — 上でcloneした場所ではない
fw init                  # Phase-0監査 + 非上書きスキャフォールド(signals/を作成)
mkdir -p memory          # 他のセクションは自分で — セクションとは単なるディレクトリ
# セクションディレクトリにマークダウンを置き
fw sync --write          # 新しさインデックス + llms.txt を再生成
fw doctor                # 鮮度 + 死んだポインタのレポート
```

**成功の判定**: `INDEX.md`にファイルがセクション別に並ぶ`AUTO-INDEX`ブロックができ、
`fw check`がexit 0。`doctor`は助言用なので、**非ゼロの終了コードはレポートであって
インストール失敗ではない。**

リポジトリはこうなる:

```
your-knowledge-repo/
├── INDEX.md      上はキュレーションされたポインタ、下は生成されたAUTO-INDEXブロック
├── AGENTS.md     initが追記したWikiプロトコルブロック — エージェント向けの契約
├── llms.txt      sync --write のたびに生成
├── memory/
└── signals/
```

`INDEX.md`でマージ衝突が起きたら: どちらかの側を取って`fw heal --write`。以上。

## あなたの組織のインスタンス

`fw init`は動くインデックスを与える。しかし*そこに何を入れるか*は教えてくれない —
その決定こそがこのWikiをあなたのものにし、どんなツールも代行できない。

このツールが前提とする対: **ハーネスは方法論とゲートを担い、Wikiは組織の文脈を担う。**
互いの代替にはならない。文脈の着地点を持たないハーネスは毎セッション同じ組織的事実を
導出し直し、ハーネスを持たないWikiは何が入るかにゲートがない。

約2ヶ月の日次運用を耐えたセクション軸(981ファイル — 上位4セクションがその73%):

| セクション | 保持するもの | 入るタイミング |
|---|---|---|
| `memory/` | 発見したセッションより長生きする事実 | 再び必要になり、コードから導出できないとき |
| `tracks/<project>/` | プロジェクト別の作業記録、蓄積 | その作業が再読の価値を生んだとき |
| `signals/` | 後で重要になる*かもしれない*観察・測定 | 気づいた時点 — 価値が出るか分かる前に記録 |
| `handoff/` | マシン・セッション・人の間で渡す状態 | 次の読み手が今の書き手でないとき |
| `audit/` | ケイデンスに沿った定期レビュー | ケイデンスが発火したとき、気が向いたときではなく |
| `digests/` | 反復的な外部スキャン | 自動ジョブが出力を落としたとき |

**2つから始めよ** — `memory/`と`signals/`。残りは、あるファイルが既存セクションに本当に
収まらない最初の瞬間に生まれさせればいい。ディレクトリ追加は安い。**誰も書き込まない
セクションは、埋めよという信号ではなく削除せよという信号だ。**

**どちらに入れるか?** 2つの問いでほぼ決まる。*これはコードから再導出できるか?* —
できるなら`memory/`に入れるものではない。*判定は出たか?* — 出ていないなら決定記録ではなく
`signals/`のエントリだ。

`examples/org-instance/`が最小の骨組みだ。**まだ`init`していない**リポジトリにコピーせよ —
自分の`INDEX.md`を持ち込むが、`fw init`は既存のINDEX.mdを上書きしない:

```sh
cp -r ~/tools/forge-wiki/examples/org-instance/. your-knowledge-repo/
cd your-knowledge-repo
fw init          # コピーされたINDEX.mdはそのまま、AGENTS.mdブロックのみ追記
fw sync --write
```

## サブコマンド

| cmd | 動作 | 失敗方向 |
|---|---|---|
| `init` | 状態監査 + スキャフォールド(決して上書きしない) | 既存ファイルがあれば拒否 |
| `sync [--write]` | AUTO-INDEXブロック再生成 | デフォルトはdry-run |
| `heal [--write]` | 衝突マーカー/重複ブロックを整理してsync | キュレーション領域の衝突は人間の仕事 |
| `lint` | フロントマターレポート | レポートのみ、決して書き換えない |
| `doctor` | 鮮度 + 死んだポインタのレポート | 助言のみ |
| `check` | インデックスドリフトゲート | ドリフト時exit 1(CIブロック) |

## ステータス

v0.1 — 約2ヶ月間毎日運用された非公開オペレータWikiから抽出: **Markdown 981ファイル、
57日中50日にコミット、マルチマシン**(2026-07-21 実測 — `git ls-files '*.md' | wc -l`;
ストア自体は非公開)。`tests/`の並行性シミュレーションがすべての堅牢性主張の回帰アンカー:
シミュレーションが測定していないことはここでは主張しない。

## 配布サーフェス

- `llms.txt` — `sync --write`ごとに生成(インデックス同様の導出物)。
- `bin/fw_mcp.py` — 依存ゼロの読み取り専用MCPサーバー(`wiki_index` / `wiki_get` /
  `wiki_search`)。読み取りはMCP、書き込みはgit + ゲート — この非対称は欠けた機能では
  なく書き込みプロトコルそのもの。
  `claude mcp add forge-wiki -- python3 <abs>/bin/fw_mcp.py <wiki_root>`

完全な契約は`SPEC.md`を参照。
