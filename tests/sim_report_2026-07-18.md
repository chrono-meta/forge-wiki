# fw.py 동시쓰기 견고성 시뮬레이션 보고서 (2026-07-18)

**대상**: `bin/fw.py` (sync/heal/check/doctor — AUTO-INDEX 블록은 first-START..last-END collapse로 자기치유 설계)
**하네스**: `tests/sim_concurrency.py` (영구 회귀앵커 — bare origin + N개 클론 프로세스 병렬, 진짜 push 레이스)
**원자료**: `tests/sim_data_2026-07-18/results_prebackoff_all.json` · `results_backoff_all.json` · `results_backoff_n50_replacement.json`
**규율**: 실패는 fudge 없이 finding으로 기록. N=50은 reps=3(+오염 rep 대체 1회).

## 핵심 결론 (3줄)

1. **콘텐츠 유실(lost-update) 0** — 전 스케일·전 rep·전 시나리오에서 "push 성공했는데 origin에서 사라진 파일" **0건**. INDEX 무결성(START 정확히 1개 · 충돌마커 0 · 히스토리 내 커밋된 마커 0)도 전 런 유지 — **충돌 시 임의 side checkout → sync 재생성** 전략이 설계대로 동작.
2. **가용성은 재시도 레시피에 좌우** — 백오프 없는 즉시 재시도는 N≥10부터 livelock(N=50: 125개 중 ~87개가 상한 10회 도달, 최종 미도달 ~45%). 지수 백오프+지터를 넣으면 N=50에서도 최종 미도달 0~16% (클린 환경 런 125/125). **fw.py 결함이 아니라 호출측 재시도 규약 문제** — 레시피를 제품 문서에 박아야 함(F-1).
3. **fw.py 자체 결함 2건 발견**(고치지 않고 기록): 고아 START 시 큐레이션 삼킴(F-2), setext 밑줄 충돌마커 오탐(F-3). 그 외 stale-index drift는 설계 일관(check가 잡고 sync 1회로 치유, F-4).

## 발견 결함 목록 (fw.py는 수정하지 않음 — 재현 커맨드 포함)

### F-1 — [호출측/문서] 재시도 규약 없이는 N≥10에서 livelock
- **증거**: prebackoff N=50 3 reps 미도달 45~49% vs 백오프 N=50 미도달 0~16%.
- **처방 후보**: `git pull` 재시도 전 `sleep(min(0.05·2^k, 2.0)·jitter)` — 하네스 워커에 구현된 레시피(`sim_concurrency.py` worker 루프)를 fw 사용 문서(SPEC 재시도 절)에 명기. fw.py 코드 결함 아님.

### F-2 — [fw.py] 고아 START(END 소실) 시: 1-pass는 START 2개 상태를 rc=0으로 통과, 2-pass는 고아 START 아래 **큐레이션 텍스트를 삼킴**
- split_regions가 END 부재 시 `auto=None`으로 판단해 블록을 **끝에 append** → 기존 고아 START 잔존(START 2개, rc=0 — 이 상태로 커밋/push 가능). 다음 sync에서 first-START..last-END collapse가 고아 START~새 블록 사이의 **큐레이션 라인까지 derived로 오분류해 소거** — "heal은 큐레이션 불가침" 독트린 위반 엣지.
- **재현**:
  ```bash
  mkdir -p /tmp/fwrepro/notes && cd /tmp/fwrepro
  printf -- '---\ndescription: n1\n---\n# n1\n' > notes/note_2026-07-01_a.md
  printf '# W\n<!-- AUTO-INDEX:START (generated by fw.py — do not hand-edit inside) -->\nstale\nIMPORTANT curated line\n' > INDEX.md
  python3 bin/fw.py sync --write   # rc=0, START 2개
  python3 bin/fw.py sync --write   # START 1개, "IMPORTANT curated line" 소실
  ```
- **수리 방향 후보**: START만 있고 END 없으면 append 대신 START~EOF를 derived로 간주하거나, 경고+rc≠0으로 HITL 요구.

### F-3 — [fw.py] setext 헤딩 밑줄(`=======`)을 충돌마커로 오탐 → warn + rc=2
- `CONFLICT_RE = ^(<{7}|={7}|>{7})`가 마크다운 setext 밑줄과 충돌. 큐레이션 영역에 있으면 "human merge required" 경고 + exit 2 → CI 게이트 오차단 가능.
- **재현**: `printf 'Title\n=======\n<!-- AUTO-INDEX:START ... -->\n<!-- AUTO-INDEX:END -->\n' > INDEX.md && python3 bin/fw.py sync --write; echo $?` → 2.
- **수리 방향 후보**: `={7}` 뒤에 나머지가 공백뿐인 git 마커는 `^={7}$`이 아니라... (git 마커는 정확히 7자+개행) — `^={7}\s*$`도 setext와 충돌하므로, START/END 사이 여부 + `<<<<<<<`/`>>>>>>>` 짝 존재로 판정하는 쪽이 견고.

### F-4 — [설계 일관·주지 필요] 클린 auto-merge 경로는 stale AUTO 블록을 push할 수 있음 (drift는 일시적)
- 충돌 없이 머지되면 재생성 없이 push됨 → origin INDEX가 실제 파일셋과 어긋남 (관측: `latest 5 of 41` vs 실제 65 — prebackoff N=50 전 rep에서 check DRIFT). `fw check`가 잡고(rc=1) `fw sync --write` 1회로 치유됨을 보존 클론에서 확인. index=derivation 독트린과 일관.
- **처방 후보**: 사용 규약을 "충돌 시에만"이 아니라 "**모든 pull/merge 후 fw sync --write**"로 문서화(비용 미미), 또는 post-merge 훅.

### 관측 (결함 아님)
- **충돌마커 히스토리 오염 0**: 전 런 `git log -S "<<<<<<<"` 0건 — 마커가 커밋된 적 자체가 없음.
- **바이너리 junk .md는 인덱스 엔트리로 등재됨**(제목 추출 실패 시 빈 요약) — 크래시 없음이 계약이고 충족. 필터링은 lint/HITL 몫.

## 측정 잔여물 (정직 기록)
- N=50 rep2 무효 사유: 측정자(이 세션)의 `pkill -f "sim_concurrency.py worker"`가 병행 중이던 스위트의 워커 35개를 오사 — 대체 rep로 보충. 별개로, 두 스위트가 겹쳐 돌던 시점의 43/50 워커 크래시 1회가 있었으나(리소스 압박 추정) stderr 미보존으로 원인 미확정 — 단독 실행 2회에서는 크래시 0. fw.py 결함 증거 아님.
- 유닛당 파일 2~3개·재시도 상한 10은 과제 스펙 고정값 — 상한을 올리면 N=50 최종 미도달이 0으로 수렴할 것으로 예상되나 **미측정**.
- 단일 관측이었던 "AUTO 블록 충돌 자기치유"는 본 시뮬로 수백 회 재현됨 (우연 아님 확정).
